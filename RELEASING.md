# Releasing

Releases are automated. Pushing a `vX.Y.Z` tag triggers
[`.github/workflows/publish.yml`](.github/workflows/publish.yml), which builds
the package, publishes it to PyPI via **Trusted Publishing** (OIDC — no tokens
stored anywhere), and creates a matching GitHub Release with autogenerated notes.

## One-time setup (PyPI Trusted Publishing)

Do this once, before the first tag-triggered release:

1. Sign in at [pypi.org](https://pypi.org/).
2. Go to **Your projects → Publishing** (or, for a brand-new name,
   <https://pypi.org/manage/account/publishing/> → *Add a pending publisher*).
3. Register a **GitHub Actions** publisher with exactly these values:
   - **PyPI Project Name:** `cascade-mcp`
   - **Owner:** `clemente-turrubiates`
   - **Repository name:** `cascade-mcp`
   - **Workflow name:** `publish.yml`
   - **Environment name:** `pypi`

No API token, no repository secret. GitHub proves the workflow's identity to PyPI
over OIDC.

> First release alternative: if you'd rather do the very first upload by hand,
> run `uvx twine upload dist/*` locally, then set up trusted publishing above so
> every subsequent release is automatic.

## Cutting a release

1. Bump `version` in [`pyproject.toml`](pyproject.toml).
2. Move the changes under `## [Unreleased]` in
   [`CHANGELOG.md`](CHANGELOG.md) into a new `## [X.Y.Z]` section and update the
   compare links at the bottom.
3. Commit both:
   ```
   git commit -am "Release vX.Y.Z"
   git push origin main
   ```
4. Tag and push:
   ```
   git tag -a vX.Y.Z -m "cascade-mcp vX.Y.Z"
   git push origin vX.Y.Z
   ```

The **Publish** workflow takes it from there: build → `twine check` → PyPI →
GitHub Release.
